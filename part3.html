How does data (state) flow in React?

Data flow in React is one-directional. It always flows down.

If a child component's state changes, does the parent get re-rendered?

No, a child's state does not affect the parent because of the unidirectional data model.

True or False? Props can be sent from the child to the parent.

False, props an only sent from the parent to the child.

True or False? A child can change or update its props.

False, props are read only values and cannot be modified by the children components.

True or False? A child component can update its local state

True, local state is completely encapsulated in the component, so it doesn't matter whether or not the component is a parent or a child.

True or False? A component does not have to be sent props.

True! Props are optional. You should only send down necessary information.

True or False? A component must keep state.

False, not only is this not necessary, but there are several advantages that display components have over stateful components.

What is the difference between state & props?

Props are read only values that are sent to a component. They are outside of the control of the component receiving them.
State represents mutatable data within a component

What is the React function that updates state?

setState

};

What is happening here?


toggleCheckbox = () => {
  this.setState(prevState => {
    return { isChecked: !prevState.isChecked };
  });
};

Why would you want to update state this way?

Here are all the different ways to call setState: 


// function signature
setState(updater[, callback])

// updater can be an object representing changes to be merged into state
resetName = () => {
  this.setState({ name: 'Oreo' });
};

// updater can also be a fn that takes the previous state as an argument
toggleCheckbox = () => {
  this.setState(prevState => {
    return { isChecked: !prevState.isChecked };
  });
};

// setState can also take an optional callback argument, which
//  is called after state has been updated
// you are sending a reference to the fn so you don't want to invoke it.
handleNameChange = event => {
  this.setState({ name: event.target.value }, this.autoSaveForm);
  
  // this would be wrong:
  // this.setState({ name: event.target.value }, this.autoSaveForm());
};

// or:
toggleCheckbox = () => {
  this.setState(prevState => {
    return { isChecked: !prevState.isChecked };
  }, this.autoSaveForm;
};

Is this a valid component?


const AuthorAvatar = props => {
  return (
    <div>
      <img src={props.authorImg} />
      <div>{props.name}</div>
      <div>{props.twitterHandle}</div>
    </div>

};

Yes! This is the way to write a stateless component as a function (functional stateless component).

What kind of event is sent to an event handler in React?

A SyntheticEvent— it works pretty much just the same as a regular event, but browser specific behavior is abstracted away for you.

Can a child change its parent's state?

This one is tricky: Yes, it is possible to do this by having the parent also send down a function to update its state. The child can then invoke that function, which will change the parent's state.

What the render process looks like for this: A call from the child updates the parent, causing the parent to re-render. Because the parent has updated, any children it has (including the one that kicked this whole process off) will also be re-rendered.

That's a lot of renders! You should avoid this pattern when possible, but it's there if necessary.

Helpful random things you might need to know

Conditionally show a component


render() {
  return (
    <div>
      {!!hasErrorMessage && <ErrorBar message="You done messed up" />}
    </div>
  );
}

If you haven't seen !! before: it casts a truthy check to a true/false value. This can be nice to use because sometimes just the truthy check (e.g., {numErrorMsgs && <ErrorBar />} ) will also display the variable you're checking.

Ternary Operator


render() {
  const { items } = this.state;
  return (
    <div>
      { /* boolean condition ? the thing to do if true : the thing to do if false; */ }
      {!!hasItems ? <DisplayComponent items={items} /> : <EmptyState />}
    </div>
  );
}

Renaming a deconstructed variable


render() {
  const { items: itemList } = this.state;
  return (
    <div>
      {!!hasItems ? <DisplayComponent items={itemList} /> : <EmptyState />}
    </div>
  );
}

{ items: itemList }
items is the name of the variable in this.state (this.state the object we're deconstructing) and itemList is the new name we want to use.

Add a displayName

The displayName field is used in debugging messages. It is helpful to set your own instead of having JSX set the value.


class TodoItem extends React.Component {
  constructor() {}
  
  static displayName = 'TodoItem';

  render() {}
}

PropTypes

A component can check that the props it receives match what it expects using PropTypes. You can also indicate whether or not a prop is required. PropTypes are checked at run time.

PropTypes also serve as great documentation.


import React from 'react';
import PropTypes from 'prop-types';

class TeddyBear extends React.Component {
  ...
}

TeddyBear.propTypes = {
  name: PropTypes.string.isRequired,
  price: PropTypes.number,
  color: PropTypes.string
};

// Alternative syntax that does exactly the same thing:
class TeddyBear extends React.Component {
  static propTypes = {
    name: PropTypes.string.isRequired,
    price: PropTypes.number,
    color: PropTypes.string
  };
}

Default Props

A component can also set default values of props it may receive. Do this where it makes sense to have defaults— for example, you should not set defaults for required props. 


import React from 'react';

// Props:
//    * name - string
//    * price - number
//    * color - string
class TeddyBear extends React.Component {
  ...
}

TeddyBear.defaultProps = {
  price: 12.99,
  color: 'brown'
};

// Alternative syntax that does exactly the same thing:
class TeddyBear extends React.Component {
  static defaultProps = {
    price: 12.99,
    color: 'brown'
  };
}

Pure functions

A function where the return value is only determined by its input values, without observable side effects. Given the same input, it will always return the same output.

This will be more important when learning Redux, but you can apply this to any React devving you do. 


// i am pure
const appendBazinga = sickBurnInput => {
  return sickBurnInput + ' Bazinga!';
};

// i am not pure
const appendBazinga = sickBurnInput => {
  sickBurnInput = sickBurnInput + ' Bazinga!';
};

Best Practices

  • Composition over Interitance: Instead of creating components that extend one another, it's more maintainable to create container components that are composed of children components. This is the pattern we've been using all through this course and is an offical React best practice.
  • Push state as high up as possible: Children components should be responsible for as little state as possible and rely on taking in props from their parent. This is also true for Flux/Redux— try to keep children components unconnected when possible and take actions as props from their parent.
  • Single source of truth: For any piece of data, only one component should "own" its value.

Testing

  • Industry standard for testing in React is Jest and Enzyme (they are 2 separate libraries).
    • Jest is kind of like a test runner library. It includes support for snapshot tests.
    • Enzyme is a library that makes rendering and manipulating components easier.
  • Sometimes you may see one but not the other (for example, Enzyme with Karma and Jasmine)
  • You may still see ReactTestRenderer and ReactTestUtils, especially in older code bases. These are the original testing libraries that React came out with.
