What is React?

  • The V in MVC
  • React is a declarative, lightweight, and flexible JavaScript library for building user interfaces.
  • It renders HTML, handles events, and manages local state
  • Developed by Facebook and open-sourced

Why use React?

  • Fast üèéÔ∏è
  • Declarative
  • Easy to understand and maintain at scale
  • Flexibility— can be used with your favorite logic/routing libraries. (But React router is great)
  • In fact, it's so flexible that it can be used on the server side as well as the client side (for isomorphic apps)!
  • And of course it can be used for cross-platform mobile development with React Native
  • Fast&emdash; hella fast bc of virtual dom & efficient state management

    • Should I use React?

    • Great for dynamic sites with lots of data
    • Also great if you want to use vanilla JS and ES6, rather than learn a new DSL (domain specific language)
    • May be overkill for very small, static sites, since it requires more explicit code than other frameworks
    • Some people prefer to keep JS and HTML separate (and prefer Vue or AngularJS) - but don't knock JSX until you've tried it!

    What React does not do

    • React by itself does not manage global state— you’ll need something like Redux or Flux for that
    • React is not a framework. It is a lightweight library (so it only comes with the basics).

    What is a Component?

    • Components are the building blocks of React
    • You connect components together to build an appliation, kind of like stacking legos
    • They are classes that extend React.Component
    • They always implement a render function, which outputs HTML

    What is a Class?

    • A class is a new Javascript feature, added in ES6 (ECMA2015).
    • It is literally a function with syntatic sugar, that behaves similarly to classes in object-oriented languages (like Java).
    • A class can have a constructor function, which is run on initialization.
    • A class can extend another class with the "extends" keyword. If it does so, it's constructor needs to use the word super().
    • We'll review all of this later!
    
		class Greeting extends React.Component {
			constructor() {
				super();
			}
		}

    What is JSX?

    • Think of it as code that looks like HTML, and is transpiled into Javascript that generates HTML.
    • (If you're curious about the X, the name is short for Javascript XML.)
    • The major advantages of JSX are: (1) it looks a lot like HTML and (2) it is more concise than writing React in pure JS
    • Making smaller, reusable components is also easier, versus other frameworks where a new template file and JS file need to be created for each new component.
    • Read more about the benefits here

    What is JSX?

    • Almost fully identical in appearance to HTML - except:
    • className - Instead of writing <li class="red">, you would write <li className="red">. This is because class is now a reserved word in Javascript.
    • key - To optimize list display performance, where lists can be reordered/changed, React asks for a key attribute that points to a unique ID for each list element.
    • Synthetic events - Some event handlers (onSubmit, onClick) have different names or implementations in different browsers. The HTML created by your code will be browser specific (onSubmit will turn into submit, etc), simplifying cross browser compatibility.
    • And some other exceptions we might get to later!

    What are transpiling and compiling?

    Compilation is the process of turning one language into another

    Transpiling is compiling between two languages that are at the same level of abstraction.

    Most browsers do not understand ES6+ syntax (or JSX), so code that uses those formats needs to be transpiled into a form of Javascript that browsers today do understand (ES5). This is usually done with BabelJS.

    You can use compilation and transpiling interchangably when talking about turning ES6/JSX into Javascript, although the term transpile is more common.

    What is the Virtual DOM?

    • First things first: the browser DOM is an in-memory abstraction of HTML
    • DOM stands for Document Object Model
    • Updating the DOM requires browser manipulation, which is slow
    • The virtual DOM is an in-memory representation of the DOM. Updating the virtual DOM does not cause the screen to be repainted, so it is much faster. Like, much faster.
  • this is the biggest reasons why other frameworks like angular and ember are slow-- they are updating the DOM more than they need to

    What is the Virtual DOM?

    We use ReactDOM.render to add a component to the VirtualDOM.

    Whenever the page loads, or data changes after a page load, React creates a new VirtualDOM, and compares it to the old one. This is called diffing. It takes the changes between them and applies them to the real DOM. This results in less browser manipulation and faster performance. 

    
		class Greeting extends React.Component {
			render() {
				return (<div>Hello World!</div>);
			}
		}
		ReactDOM.render(<Greeting />, document.getElementById('Greeting'));

    The first argument to ReactDOM.render() is the JSX to render, and the second is an element on the page to insert it in (selected with vanilla JS).

    Note that we don't need the constructor method yet, but we'll start using it later.

  • for example, if you have a todo list of 10 items, and you edit one item, only the DOM related that single item would be updated. in angular, e.g. the whole list would be updated.

    Quick note on object destructuring

    Object destructuring is a new thing in ES6+. You will often see it used in React code because it can make your code less verbose.

    For example: you might see something like this. 

    
const { name, breed } = this.props;
console.log('name: ', name);
console.log('breed: ', breed);

    What this is doing is creating two new variables, name and breed, by pulling the values from this.props. We can then refer to those variables without needing to put this.props in front of them. Same for state.

    Note that the variables created by destructuring are only available in the context of whatever scope you're in.

    What is state?

    • You can think of state as what everything is set to in your application at a particular moment in time.
    • Your application does or displays things in a certain way given a certain state.
    • There are two different kinds of state: Global and Local
    • Global state can be seen, and possibly manipulated, by multiple components in your application.
    • Local state can only been seen and manipulated by the component that state is local to.
    • Local state is initialized in the constructor function
    • Global state would be managed by something like Flux or Redux.
    • Local state is managed by React.

    What is state?

    Local state in a parent component can be passed down to its childen as props, as we saw in the Greeting example:

    
		class Greeting extends React.Component {
			render() {
				return (<div>{this.props.message}</div>);
			}
		}
		class App extends React.Component {
			constructor() {
				super();
				this.state = {

				};

			render() {

    State Management in React

    • Data flow in React is unidirectional. You’ll also hear this said as: React has one-way data binding.
    • What this means is that a child component's state doesn't affect the parent's (although the child can use a function to update its parent's component, as we will see in a moment)
    • This means that there is an explicit hierarchy in your application, where parent components manage one or several smaller child components by sending them information through props.
    • This leads to fewer re-renders and thus is one of the key aspects of React that makes it performant.
    • This is also the thing that makes React less intuitive than, say, Angular, which as two-way data binding. But this will make more sense as we work through more examples and projects.

    Updating the parent's state through the child

    To have a child component update the parent's state, we'll need the parent to pass in a function to the child through props 

    
		class Greeting extends React.Component {
			render() {
				return (<div onClick={this.props.changeMessage.bind(this)}>{this.props.message}</div>);
			}
		}
		class App extends React.Component {
			constructor() {
				super();
				this.state = {
					message: 'Hello World!'
				};
			}
			render() {
				return (<Greeting message={this.state.message} changeMessage={this.sayGoodbye} />);
			}
			sayGoodbye(event) {
				event.preventDefault();
				Equivalent to this.setState({ message: 'Goodbye' })
			}
		}
		ReactDOM.render(<App />, document.getElementById('app'));

    Code Breakdown!

    Let's go over the last example piece by piece:

    • We added a sayGoodbye method to the App class, and used it as an event handler in the Greeting class. Event handlers are functions called from onClick, onSubmit, etc. 
      
		sayGoodbye(event) {
			event.preventDefault();
			this.setState({message: 'Goodbye'});
		}
    • Event handlers do not need to be invoked (have parentheses at the end). But when a click happens, the event handler is called with an event as the argument. To prevent side effects (like form submissions from pressing a button in a form), use event.preventDefault() (above). 
      
		render() {
			return (<div onClick={this.props.changeMessage.bind(this)}>{this.props.message}</div>);
		}

    Relationship b/w State and Props

    • Props represent state from a parent component. They are sent to the child(ren) as read-only values.
    • Props can be used to initialize a child's state. For example, in an edit form.
    • Each component, parent or child, is free to update its own state. When a parent's state changes, its child(ren) will get re-rendered.*
    • * Caveat: There are ways to fine-tune re-renders through lifecycle methods, which we'll talk about a bit later.
    Let's be more explicit about the relationship between state and props.

    ES6+: Arrow functions

    Arrow functions are a new, improved syntax for functions

    
		// Simplified (for when there's on argument and only one call in the function body)
		this.state.messages.map(message =>
						(<span key={message.id}>message.value</span>)
		);
		// More common
		this.state.messages.map((message, index) => {
			console.log(index);
			return (<span key={message.id}>message.value</span>);
		});
		// ES5 equivalent
		this.state.messages.map(function(message) {
			return (<span key={message.id}>message.value</span>)
		});

    One benefit of using arrow functions is that they do not define their own `this`, so you don't need to bind this anymore! (var that=this, anyone?) And they are more concise than ES5 functions. Yay!
